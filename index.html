正文 · Build Your AI PM OS

P1写在前面

1.1不是提示词技巧课，而是工作系统课

大家好，今天分享的主题是 AI PM Builder OS——把 AI 装配成产品经理的长期工作系统，副标题是「如何借助 AI 进行产品工作提效」。

先把这次分享的边界讲清楚：这不是一场提示词技巧课，也不是工具安利课。我们不只讨论怎么让 AI 写一段更好看的文案，而是讨论产品经理如何把 AI 变成一个能长期协作的工作系统（合作伙伴 / 个人助理）。

产品经理的日常工作很复杂：要看资料、懂业务、写需求、做原型、跟研发沟通、盯上线、做复盘。AI 真正有价值的地方，是把这些环节串起来，让我们更快进入判断和决策，而不是陷在重复劳动里。

P2培训目标与核心结论

2.1培训目标

完成培训后，参与者应该能做到：

  1. 用 AI 快速完成资料检索、信息整理、需求分析、PRD 起草、原型迭代、评审准备和上线验收。
  2. 把一次性对话沉淀为可复用的工作流、提示词、模板、项目说明和评测标准。
  3. 理解 豆包、ChatGPT、Gemini、Cursor、Claude Code、OpenAICodex、百智/Obsidian、MCP、Skills 在产品工作里的不同定位。
  4. 能用"输入材料 + 工作目录 + 任务目标 + 验收标准 + 风险边界"的方式指挥 AI 工作。
  5. 能从"AI 帮我写内容"升级为"AI 帮我推进产品交付"。

2.2核心结论

第一，AI 提效的核心不是生成内容，而是推进交付。只会让 AI 写文档，效率提升是有限的；能让 AI 帮你读资料、拆任务、生成原型、检查风险、沉淀复盘，才是真正的提效。

第二，产品经理要从"会聊天"升级为"会指挥"。AI 不是你问一句它答一句，而是你给它上下文、边界、目标和验收标准，它按任务推进。

第三，好的 AI 协作需要五件套：工作目录、背景材料、任务说明、验收标准和留痕机制。

第四，AI 是 PM 的协作团队，不是 PM 的替身。AI 能帮我们处理大量结构化、重复性、信息密集型工作，但用户价值、业务取舍、优先级和风险判断仍然要由 PM 负责。

第五，团队级提效靠模板和规范。个人经验可以让一个人变快，但团队要整体变快，必须有统一目录、提示词模板、评审标准和复盘机制。

P4我的 AI Agent teams

4.1两条理解轴

AI 工具不按品牌背诵，而是按两条轴来理解：

  1. 从辅助到交付
    有的工具只是帮你想、帮你写、帮你总结；有的工具已经能真正进入项目、改文件、跑任务、交付结果。
  2. 从零代码到工程化执行
    有的工具更适合无代码、设计到代码、快速原型；有的工具更适合专业开发、复杂项目和长期迭代。

4.2所以，我的 AI Agent teams 不是一个工具，而是 5 层协作体系

层级代表工具本质定位我怎么用
通用对话层 豆包 / Gemini / ChatGPT / 通用 AI 助手 / 模型入口 做脑暴、资料总结、轻量分析、快速问答
原型生成层 Figma AI / Google AI Studio / Claude Design /Google stitch 零代码原型与设计到代码工具 把想法尽快变成可讨论的页面、流程和高保真原型
执行开发层 Cursor / Claude Code / OpenAI Codex /WorkBuddy 执行型 Agent / IDE 强在进入真实项目、读文件、改文件、跑命令、执行脚本、推动结果落地
跑通 Demo 层 GitHub / Vercel / Supabase 代码托管、部署与后端服务层 让原型真正上线、可访问、可登录、可存数据
知识沉淀层 Obsidian / Notion / 百智 知识库 / 上下文层 沉淀提示词、案例、复盘、模板和项目索引
打个小广告 📣 就一句，不占用主线～
百智 垂直业务 执行型 Agent AI Native 知识生产

百智——更偏 垂直业务的 执行型Agent （AI Native 知识生产平台），核心定位不是通用聊天，而是以录音采集为入口、以 PC 端深度研究与成果交付为核心，把 AI 能力封装进完整业务链路中的工作平台。

4.3Skill 和 MCP 到底是什么关系

  1. Skill
    不只是"提示词模板"，更像一套可复用的任务能力包，里面可以包含：方法说明、执行规则、步骤清单、参考资料，甚至脚本和命令。它解决的是"这类任务通常该怎么做"。
  2. MCP
    更像"外部工具和数据接口层"，让 Agent 能连接搜索、数据库、浏览器、Notion、业务系统、第三方服务和 API。它解决的是"Agent 能接到哪些外部能力"。
  3. Agent
    负责理解目标、拆任务、判断什么时候调用哪个 Skill、什么时候调用哪个 MCP，并在需要时真正执行动作，比如读文件、改文件、调工具、跑命令、执行脚本、返回结果。
  4. 知识库
    负责把方法、模板、案例、复盘和产物沉淀下来，形成长期上下文和组织记忆，下一次可以直接复用。

P5AI 产品经理如何与 Agent 协作共创

工具只是载体，真正决定结果质量的，是产品经理和 Agent 的关系，不是"提需求 -> 等答案"，而是"提问题 -> 补背景 -> 辩方案 -> 压范围 -> 共创结果"。

5.1一次高质量共创通常会经历 5 轮

  1. 第一轮：抛出任务
    先告诉 Agent 你要解决什么问题。
    比如：做一个百销产品原型，或输出一份给研发评审的一期 PRD。
  2. 第二轮：补业务背景
    告诉它这是谁的需求、解决什么业务问题、为什么现在做、上下游是谁、成功标准是什么。
    如果背景不清，再强的 AI 也只能局部猜测。
  3. 第三轮：澄清模糊点
    Agent 会反过来暴露问题：边界不清、字段不够、流程断裂、角色冲突、目标不一致。
    很多时候，AI 最有价值的地方不是直接给答案，而是逼你把需求讲明白。
  4. 第四轮：互相辩论和纠偏
    产品经理不是被动接收答案，而是要不断反驳、压范围、改优先级。
    比如：
    • 这个方案太像后台，不像一线业务使用界面
    • 这个流程太长，一期做不完
    • 这个交互很完整，但没有解决最关键的转化问题
  5. 第五轮：收敛成结果
    最后才进入文档、原型、页面、代码、验收标准这些交付物。
    结果不是"AI 一次生成的"，而是"产品经理和 Agent 多轮共创出来的"。
本节小节：
产品经理和 Agent 的关系，不是"提需求 -> 等答案"，而是"提问题 -> 补背景 -> 辩方案 -> 压范围 -> 共创结果"。

5.2AI 产品工作五件套

  1. 工作目录 · 让 AI 先读懂你的世界

    AI 必须知道资料、代码、原型、知识库分别在哪里。如果没有工作目录，AI 每次都像新同事入职；如果有目录和规则，AI 每次都能从组织记忆开始工作。

    很多 AI 产出不稳定，不是模型差，而是它不知道你的项目结构、不知道哪些文件不能改、不知道真实资料在哪里。

    例如当前 Playground 工作区：projects/ 放项目，clients/百融百智/ 放业务资料，obsidian/ 放知识沉淀，ai/ 放工具说明。

    目录用途
    projects/可运行项目
    clients/百融百智/业务资料、原型、提示词、研究
    obsidian/知识库、索引、复盘、方法论
    ai/AI 工具上下文和目录说明
    archive/历史备份
  2. 背景材料
    产品工作最怕凭空生成。
    AI 应该先读已有 PRD、原型、会议纪要、竞品资料，再开始输出。
  3. 任务说明
    不只是写"帮我做什么"，而是要把目标用户、业务背景、关键场景、输出对象、上下游关系和约束讲清楚。
    很多产品任务做不准，不是 AI 不够强，而是我们没有把业务逻辑链路交代完整。
    只要背景没说明白、逻辑没捋清，再强的 AI 也只能局部猜测，最后产出往往"看起来很完整"，但并不真正可用。
  4. 验收标准
    要明确格式、完整性、测试项和风险项。
    这些标准越清晰，AI 的产出越接近真实交付。
  5. 留痕机制
    一次对话结束后，提示词、版本、产物和复盘都要沉淀下来。
    否则下次又要从零开始。
本节小结：
AI 提效不是一句提示词，而是一套有上下文、有边界、有验收、有沉淀的工作系统。

5.3人机协作边界

AI 适合
  1. 批量读取和整理资料
  2. 生成结构化初稿
  3. 提醒遗漏和风险
  4. 转换格式和口径
  5. 做重复性检查
PM 必须负责
  1. 用户价值判断
  2. 业务优先级
  3. 关键取舍
  4. 事实核验
  5. 对外承诺和最终结论

5.4要建立的使用认知

  1. AI 不是速记员，也不是你说一句它写一句。
  2. AI 也不是绝对权威，它会给出很多看似合理、其实不符合业务现实的方案。
  3. Agent 最强的价值，在于帮助 PM 暴露模糊点、发现冲突、提出备选方案、推动结果收敛。
  4. 产品经理的核心能力没有消失，只是从"亲自写所有内容"转成了"组织问题、提供判断、推动共创"。

P7收尾

总结：

我更愿意这样概括产品经理的「AI 能力」：本质不在会不会写提示词，而在能否完成三次转化——把模糊需求落成清晰任务，把散乱资料织成可信上下文，把一次性产出沉淀为个人/团队资产。

与此同时，也请守住你用时间与案例养出来的用户 sense、商业 sense、审美与品味、独立判断，以及系统化的结构化思维。模型与工具可以日更，但这些无法被一键安装；无论技术周期如何切换，它们才是你真正属于自己的护城河，也是最难被复制的竞争壁垒。

下一步建议：

  1. 每个 PM 选一个真实工作任务做 AI 改造
  2. 每周沉淀 1 个高质量提示词或工作流
  3. 每月复盘 1 次 AI 提效案例，形成团队模板库

AI 产品工作流治理规范

信息安全、事实核验、PRD 与原型治理、上线规范、产物评审 Rubric。

1. 信息安全边界

禁止输入或上传：

  1. 真实 API Key、数据库密码、token。
  2. 未脱敏客户个人信息。
  3. 未授权合同和商业敏感条款。
  4. 公司内部未公开财务数据。
  5. 生产环境 .env.local。

允许在安全范围内使用：

  1. 脱敏后的业务样例。
  2. 已公开的竞品资料。
  3. 内部可共享的 PRD 和原型。
  4. 经过授权的客户资料。
  5. 本地项目结构和非敏感代码。

2. 事实核验规范

必须核验的内容：

  1. 法规、政策、合规结论。
  2. 价格、套餐、商业条款。
  3. 竞品最新功能。
  4. 市场规模和行业数据。
  5. 对外发布文案中的事实表述。
  6. 技术可行性和上线配置。

来源记录最低要求：

  1. 来源名称。
  2. 完整 URL 或本地文件路径。
  3. 查询时间。
  4. 数据所在章节或页面位置。
  5. 是否为 AI 推断。

3. PRD 和原型治理

3.1 PRD 产出检查

PRD 必须包含：

  1. 背景和目标。
  2. 目标用户和场景。
  3. 本期范围。
  4. 不做范围。
  5. 用户流程。
  6. 页面结构。
  7. 字段说明。
  8. 权限规则。
  9. 异常状态。
  10. 验收标准。
  11. 风险和待确认问题。

3.2 原型产出检查

原型必须说明：

  1. 适用端：PC、移动端或响应式。
  2. 核心入口。
  3. 页面跳转。
  4. 表单和提交动作。
  5. 列表和详情页。
  6. 数据是否为 mock。
  7. 哪些交互只是演示。
  8. 后续真实产品需要补什么。

4. AI 产物评审 Rubric

维度检查问题
完整性是否覆盖任务要求和关键场景
准确性是否有事实错误或未经验证结论
结构性是否便于评审、执行和交接
可落地性是否有字段、流程、验收和边界
风险控制是否暴露不确定性和待确认项
可复用性是否沉淀为模板或工作流
